Abstract

Autonomous AI agents require persistent memory that transcends session boundaries, yet no analogue of the operating-system filesystem has been formalized for agent cognition. Traditional filesystems store untyped byte streams; agent memory demands typed schemas, causal versioning, semantic search, and graph-structured lineage. We introduce a categorical memory model in which memory is a functor $M\colon \mathcal{S} \to \mathcal{S}\!t$ from a schema category $\mathcal{S}$ (whose objects are typed schemas and morphisms are schema transformations) to a storage category $\mathcal{S}\!t$ (whose objects are storage backends and morphisms are queries). Type safety is expressed as natural transformations between memory functors; versioning is a temporal functor $V\colon \mathsf{Timeline} \to \mathcal{S}\!t$; and graph lineage is a 2-categorical structure layered over the base categories. We prove that the generic typed wrapper $\mathsf{Mem}[S]$ is a representable functor (a Yoneda-lemma connection), that type structure is preserved under storage round-trips, and that the memory operations evolve, merge, and split form a monad on the memory category. The reference implementation, ContextFS, realizes this framework with 24 memory-type variants, a multi-backend storage router (SQLite, ChromaDB, FalkorDB, PostgreSQL), vector clocks for causal consistency, content hashing for deduplication, and an MCP server for universal client access. We provide a production-quality Elixir/OTP implementation using ETS for working memory and Mnesia for persistent knowledge, demonstrating that the categorical abstractions yield both formal guarantees and practical performance.

Keywords: agent memory, typed filesystem, category theory, knowledge representation, persistent cognition, OTP, MCP, semantic search

1 Introduction

The first two installments of this series established the kernel architecture and the scheduler for an AI Operating System. Yet an operating system without a memory subsystem is merely a stateless dispatcher: it can schedule computations but cannot accumulate knowledge across them. Every traditional OS dedicates a substantial fraction of its complexity to memory management—virtual memory, page tables, filesystem hierarchies, caching layers—because persistent, organized storage is the foundation upon which all higher-level abstractions rest.

For autonomous AI agents, the memory problem is simultaneously more constrained and more ambitious than its classical counterpart. More constrained because agents do not need byte-addressable random access to flat address spaces; more ambitious because agent memory must support:

  1. Typed schemas: each memory entry conforms to a domain-specific type, not a raw byte stream.
  2. Causal versioning: memory evolves over time, and the system must track why each change occurred (observation, inference, correction, decay).
  3. Semantic search: retrieval by meaning, not merely by address or filename.
  4. Graph lineage: memories relate to one another through typed edges—one memory may supersede, contradict, reference, or evolve into another.
  5. Multi-backend coordination: different storage engines excel at different query patterns (relational, vector, graph), and the memory layer must route transparently.

Traditional filesystems fail on every count. POSIX provides open/read/write/close on untyped byte sequences organized in a tree hierarchy. Plan 9's union mounts and synthetic filesystems introduced compositional namespace ideas but remained fundamentally untyped. Semantic filesystems added metadata-based retrieval but lacked formal type guarantees. None of these systems address versioning with causal annotations, graph-structured relationships, or the combination of exact and approximate (embedding-based) retrieval that agent cognition requires.

This paper formalizes agent memory using category theory and provides a reference implementation called ContextFS. The categorical framework yields three concrete benefits:

  1. Compositionality: memory operations compose cleanly because they are morphisms in well-defined categories.
  2. Type safety: the functor $M\colon \mathcal{S} \to \mathcal{S}\!t$ preserves type structure, and we prove round-trip fidelity.
  3. Extensibility: new schema types, storage backends, and relationship kinds can be added as new objects and morphisms without modifying the core framework.

The structure of the paper is as follows. Section 2 surveys relevant background. Section 3 develops the formal memory model. Section 4 details the type system. Section 5 formalizes versioning and lineage. Section 6 presents the multi-backend storage architecture. Section 7 establishes reliability guarantees. Section 8 provides the Elixir/OTP implementation. 2 Background

2.1 Virtual Memory and Filesystems

The virtual memory subsystem of a modern OS provides each process with an isolated address space mapped onto physical memory and secondary storage through page tables. The key abstraction is indirection: processes reference virtual addresses, and the MMU translates these to physical locations. This indirection enables demand paging, copy-on-write, memory-mapped files, and shared memory regions.

Filesystems layer a persistent namespace on top of block devices. The Unix filesystem introduced the inode/directory model: files are sequences of bytes identified by inodes, organized in a tree of directories. The Virtual Filesystem Switch (VFS) in Linux provides a uniform interface over heterogeneous implementations—ext4, XFS, NFS, FUSE—through a common set of operations (inode_operations, file_operations, super_operations).

Plan 9 radicalized the "everything is a file" philosophy: network connections, process state, graphics, and device control are all accessed through filesystem operations on synthetic file servers. These designs are profoundly influential, yet they share a fundamental limitation: files are untyped byte streams. The filesystem knows nothing about the structure of what it stores.

2.2 Knowledge Representation and Semantic Memory

Frame systems organize knowledge around stereotypical situations with slots and defaults. Semantic networks represent concepts as nodes and relationships as labeled edges. Description logics provide formal semantics with decidable reasoning for ontological knowledge. Knowledge graphs have become the dominant paradigm, exemplified by Wikidata and Google's Knowledge Graph. The connection to agent memory is direct: an agent's accumulated knowledge forms a personal knowledge graph, with the crucial addition that this graph evolves over time.

2.3 Embedding Spaces and Semantic Search

Transformer-based language models produce dense vector representations (embeddings) that capture semantic similarity. Retrieval-augmented generation (RAG) exploits this by storing text chunks as vectors and retrieving relevant context at query time through approximate nearest neighbor (ANN) search. For agent memory, embeddings provide retrieval by meaning—analogous to human associative memory, where recall is triggered by semantic similarity rather than explicit addressing.

2.4 Category Theory Preliminaries

A category $\mathcal{C}$ consists of objects $\mathrm{ob}(\mathcal{C})$, morphisms $\mathrm{mor}(\mathcal{C})$ with source and target maps, an associative composition, and identity morphisms. A functor $F\colon \mathcal{C} \to \mathcal{D}$ maps objects to objects and morphisms to morphisms, preserving composition and identities. A natural transformation $\eta\colon F \Rightarrow G$ between parallel functors is a family of morphisms $\eta_c\colon F(c) \to G(c)$ satisfying the naturality square.

The Yoneda lemma states that for any functor $F\colon \mathcal{C}^{\mathrm{op}} \to \mathbf{Set}$ and object $c \in \mathcal{C}$:

$$\mathrm{Nat}(\mathrm{Hom}_{\mathcal{C}}(-,c), F) \cong F(c)$$

A functor is representable if it is naturally isomorphic to $\mathrm{Hom}_{\mathcal{C}}(-,c)$ for some object $c$. A monad on $\mathcal{C}$ is an endofunctor $T\colon \mathcal{C} \to \mathcal{C}$ equipped with natural transformations $\eta\colon \mathrm{id}_{\mathcal{C}} \Rightarrow T$ (unit) and $\mu\colon T^2 \Rightarrow T$ (multiplication) satisfying associativity and unit laws. A 2-category extends a category by adding 2-morphisms with two composition operations satisfying interchange laws.

3 Memory Model

3.1 The Schema Category

Definition 3.1 (Schema Category $\mathcal{S}$)

The schema category $\mathcal{S}$ has:

  • Objects: typed schemas $S_1, S_2, \ldots$, each specifying a set of typed fields with validation constraints. In ContextFS, these include BaseSchema and its 24 domain-specific descendants.
  • Morphisms: schema transformations $f\colon S_i \to S_j$, which are structure-preserving maps between schemas.
  • Identity: the identity transformation $\mathrm{id}_{S}\colon S \to S$.
  • Composition: given $f\colon S_1 \to S_2$ and $g\colon S_2 \to S_3$, the composite $g \circ f\colon S_1 \to S_3$.

The schema category has a rich subcategory structure:

$$\mathcal{S} = \mathcal{S}_{\mathrm{core}} \cup \mathcal{S}_{\mathrm{ext}} \cup \mathcal{S}_{\mathrm{wf}} \cup \mathcal{S}_{\mathrm{meta}} \cup \mathcal{S}_{\mathrm{sys}}$$
Remark

The BaseSchema in ContextFS serves as an initial object in $\mathcal{S}$ in the sense that every domain schema admits a unique embedding from BaseSchema (via inheritance). The _schema_name class variable acts as a tagging morphism that distinguishes objects.

3.2 The Storage Category

Definition 3.2 (Storage Category $\mathcal{S}\!t$)

The storage category $\mathcal{S}\!t$ has:

  • Objects: storage backends $B_1, B_2, \ldots$, each providing capabilities (save, recall, search, delete, update). In ContextFS: SQLite, ChromaDB, FalkorDB, PostgreSQL.
  • Morphisms: queries $q\colon B_i \to B_j$, representing data-flow operations.
  • Identity: the no-op query $\mathrm{id}_B$.
  • Composition: pipeline composition of queries.

Each backend $B$ comes with a capability set $\mathrm{Cap}(B) \subseteq \{\texttt{save}, \texttt{recall}, \texttt{search}, \texttt{delete}, \texttt{update}, \texttt{vector\_search}, \texttt{graph\_traverse}\}$, defining a functor $\mathrm{Cap}\colon \mathcal{S}\!t \to \mathcal{P}(\mathrm{Ops})$.

3.3 The Memory Functor

Definition 3.3 (Memory Functor)

The memory functor $M\colon \mathcal{S} \to \mathcal{S}\!t$ maps:

  • Each schema $S \in \mathrm{ob}(\mathcal{S})$ to a storage representation $M(S) \in \mathrm{ob}(\mathcal{S}\!t)$.
  • Each schema transformation $f\colon S_1 \to S_2$ to a storage operation $M(f)\colon M(S_1) \to M(S_2)$.

subject to: $M(\mathrm{id}_S) = \mathrm{id}_{M(S)}$ and $M(g \circ f) = M(g) \circ M(f)$.

    S₁ ----f----> S₂
    |              |
    M              M
    |              |
    v              v
  M(S₁) --M(f)--> M(S₂)

3.4 Representability and the Yoneda Connection

Theorem 3.1 ($\mathsf{Mem}[S]$ is Representable)

The functor $\mathsf{Mem}[-]\colon \mathcal{S}^{\mathrm{op}} \to \mathbf{Set}$ is representable. There exists a universal schema $U \in \mathrm{ob}(\mathcal{S})$ such that:

$$\mathsf{Mem}[-] \cong \mathrm{Hom}_{\mathcal{S}}(-, U)$$

The representing object $U$ is the BaseSchema equipped with all optional extensions.

Proof.

By the Yoneda lemma, $\mathrm{Nat}(\mathrm{Hom}_\mathcal{S}(-, U), \mathsf{Mem}[-]) \cong \mathsf{Mem}[U]$. Let $U = \texttt{BaseSchema}$ with all field types as optional. For any schema $S$ and instance $m \in \mathsf{Mem}[S]$, the unique morphism $\phi_m\colon S \to U$ embeds the instance into the universal schema. This defines a natural bijection:

$$\mathsf{Mem}[S] \ni m \longmapsto (\phi_m, m|_{\mathrm{base}}) \in \mathrm{Hom}_\mathcal{S}(S, U) \times \mathsf{Mem}[U]_{\mathrm{base}}$$

Since every schema transformation factors through $U$ via the respective embeddings, and the functor respects this factorization by the type-preservation property, the natural transformation is an isomorphism. The Yoneda embedding ensures faithfulness.

□
Corollary 3.2

The set of all memory instances of schema type $S$ is fully determined by the morphisms from $S$ into the universal schema $U$:

$$|\mathsf{Mem}[S]| = |\mathrm{Hom}_\mathcal{S}(S, U)|$$

3.5 Versioning as Temporal Functor

Definition 3.4 (Timeline Category)

The timeline category $\mathsf{Timeline}$ has timestamps as objects and a unique morphism $t_i \to t_j$ whenever $t_i \leq t_j$ (a thin category / preorder).

Definition 3.5 (Versioning Functor)

The versioning functor $V\colon \mathsf{Timeline} \to \mathcal{S}\!t$ maps each timestamp $t$ to the state of storage at time $t$. The versioned memory wrapper is the composite:

$$\mathsf{VMem}[S] = V \circ M|_S \colon \mathsf{Timeline} \to \mathcal{S}\!t$$

4 Type System

4.1 Schema-Indexed Types

Definition 4.1 (Schema-Indexed Memory)

For a schema $S \in \mathrm{ob}(\mathcal{S})$, the type $\mathsf{Mem}[S]$ provides:

  • $\mathsf{Mem}[S].\mathtt{create}(d\colon S) \to \mathsf{Mem}[S]$: construct a new memory wrapping validated data.
  • $\mathsf{Mem}[S].\mathtt{wrap}(d\colon S) \to \mathsf{Mem}[S]$: wrap existing data with fresh identifier and timestamp.
  • $\mathsf{Mem}[S].\mathtt{data} \to S$: extract the typed payload.
  • $\mathsf{Mem}[S].\mathtt{id} \to \mathrm{UUID}$: unique identifier.
  • $\mathsf{Mem}[S].\mathtt{schema\_name} \to \mathrm{String}$: the schema's type tag.

Type safety is enforced at two levels: compile-time (via dialyzer/typespecs in Elixir) and runtime (via struct enforcement on every create, wrap, and update operation).

4.3 Type-Safe Operations as Natural Transformations

Definition 4.2 (Storage Natural Transformation)

Let $\mathrm{InMem}, \mathrm{OnDisk}\colon \mathcal{S} \to \mathbf{Set}$ be functors mapping each schema $S$ to the set of in-memory and on-disk instances respectively. The save operation is a natural transformation:

$$\mathtt{save}\colon \mathrm{InMem} \Rightarrow \mathrm{OnDisk}$$

satisfying the naturality condition for every schema morphism $f\colon S_1 \to S_2$.

  InMem(S₁) --save_S₁--> OnDisk(S₁)
      |                        |
  InMem(f)               OnDisk(f)
      |                        |
      v                        v
  InMem(S₂) --save_S₂--> OnDisk(S₂)
Theorem 4.1 (Type Preservation Under Round-Trip)

For any schema $S \in \mathrm{ob}(\mathcal{S})$ and memory instance $m \in \mathsf{Mem}[S]$:

$$\mathtt{recall}_{S}(\mathtt{save}_{S}(m)) \cong m$$

The composite $\mathtt{recall} \circ \mathtt{save}\colon \mathrm{InMem} \Rightarrow \mathrm{InMem}$ is naturally isomorphic to $\mathrm{id}_{\mathrm{InMem}}$.

Proof.

Bijection: save serializes $m$ using schema-specific serializers. The content hash $h(m)$ is stored alongside. recall deserializes using the same schema $S$ (resolved via SchemaRegistry). Since serialization and deserialization are inverse operations for the validators, the round-trip is the identity up to isomorphism.

Naturality: For a schema morphism $f\colon S_1 \to S_2$ and $m \in \mathrm{InMem}(S_1)$:

$$\mathtt{recall}_{S_2}(\mathrm{OnDisk}(f)(\mathtt{save}_{S_1}(m))) = \mathtt{recall}_{S_2}(\mathtt{save}_{S_2}(\mathrm{InMem}(f)(m))) = \mathrm{InMem}(f)(m)$$
□

4.4 The Schema Registry

The SchemaRegistry provides runtime type resolution. Categorically, it is a section of the forgetful functor $U\colon \mathcal{S} \to \mathbf{Set}$:

$$\texttt{SchemaRegistry}\colon \mathrm{Names} \hookrightarrow \mathcal{S}, \qquad U \circ \texttt{SchemaRegistry} = \mathrm{id}_{\mathrm{Names}}$$

This section is injective by construction (schema names are unique), ensuring unambiguous type resolution at runtime.

5 Versioning and Lineage

5.1 The Reason Category

Definition 5.1 (Reason Category $\mathsf{Reason}$)

The reason category $\mathsf{Reason}$ has:

  • Objects: change reasons {Observation, Inference, Correction, Decay}.
  • Morphisms: subsumption relations. Correction subsumes Observation; Inference extends Observation; Decay is terminal.
defmodule MemoryLayer.ChangeReason do
  @type t :: :observation | :inference | :correction | :decay

  def subsumes?(:correction, :observation), do: true
  def subsumes?(:inference, :observation), do: true
  def subsumes?(:decay, _), do: true
  def subsumes?(x, x), do: true
  def subsumes?(_, _), do: false
end

5.2 Versioned Memory

Definition 5.2 (Versioned Memory)

For a schema $S$, the versioned memory $\mathsf{VMem}[S]\colon \mathsf{Timeline} \to \mathbf{Set}$ maps each timestamp $t$ to the version at time $t$, consisting of: the typed data $d_t \in S$, the change reason $r_t \in \mathrm{ob}(\mathsf{Reason})$, the version number $v_t \in \mathbb{N}$, and the parent version reference.

The version chain forms a path:

$$m_{t_0} \xrightarrow{r_1} m_{t_1} \xrightarrow{r_2} m_{t_2} \xrightarrow{r_3} \cdots$$

The edge relation category $\mathsf{Edge}$ has memory instances as objects and typed edges as morphisms, drawn from: {EVOLVED_INTO, REFERENCES, SUPERSEDES, CONTRADICTS, SUPPORTS, DERIVED_FROM, PART_OF, TRIGGERS, BLOCKED_BY}.

5.4 Memory Operations as a Monad

Proposition 5.1 (Memory Operations Monad)

The operations evolve, merge, and split define a monad $(T, \eta, \mu)$ on the category of memory instances $\mathcal{M}$:

  • $T\colon \mathcal{M} \to \mathcal{M}$ wraps a memory in a "pending operation" context.
  • $\eta\colon \mathrm{id}_\mathcal{M} \Rightarrow T$ embeds an unchanged memory as a trivial pending operation.
  • $\mu\colon T^2 \Rightarrow T$ flattens nested pending operations into a single one.
Proof.

Left unit: $\mu \circ T\eta = \mathrm{id}_T$. Starting with $T(m)$, wrapping in a trivial outer context and flattening recovers $T(m)$.

Right unit: $\mu \circ \eta T = \mathrm{id}_T$. Embedding via $\eta$ at the outer level and flattening gives back $T(m)$.

Associativity: $\mu \circ T\mu = \mu \circ \mu T$. For $T^3(m)$, flattening the inner two then the result equals flattening the outer two then the result, because operation composition is associative—change reasons accumulate as a list and list concatenation is associative.

Concretely: evolve(m, r) creates a new version (this is $T$ applied with context $r$); merge(m1, m2) is the tensor product in the monoidal structure; split(m) is the comonadic dual, and together they provide a bimonad structure.

□
Remark

The merge and split operations extend the monad to a bimonad. The full bimonadic structure ensures that agent memory supports both accumulation (merge) and decomposition (split) of knowledge, essential for multi-agent collaboration.

6 Storage Architecture

6.1 Multi-Backend Router as Coproduct

Definition 6.1 (Storage Router)

The storage router is the coproduct of backend objects in $\mathcal{S}\!t$:

$$\mathrm{Router} = \coprod_{i} B_i = B_{\mathrm{SQLite}} \sqcup B_{\mathrm{Chroma}} \sqcup B_{\mathrm{Falkor}} \sqcup B_{\mathrm{Postgres}}$$

equipped with injection morphisms $\iota_i\colon B_i \to \mathrm{Router}$ and a universal property.

Algorithm 1: StorageRouter dispatch
Input: Operation op, Schema S, Data d
Output: Result r
backends ← route(S, op)
for each B ∈ backends do
if op ∈ Cap(B) then
r_B ← B.execute(op, S, d)
r ← merge_results({r_B})
return r

The routing policy implements fan-out/fan-in: Save writes to all relevant backends; Recall reads from the primary backend; Search dispatches to the best-suited backend.

6.2 Semantic Search

Definition 6.2 (Embedding Functor)

The embedding functor $E\colon \mathcal{S} \to \mathbf{Vect}_\mathbb{R}$ maps each schema $S$ to $\mathbb{R}^d$:

$$E(S) = \mathrm{embed}(\mathrm{text}(S)) \in \mathbb{R}^d$$

Semantic search becomes cosine similarity computation:

$$\mathtt{search}(q, k) = \mathrm{top}_k\left\{ m \in \mathsf{Mem}[-] \;\middle|\; \frac{\langle E(q), E(m) \rangle}{\|E(q)\| \cdot \|E(m)\|} > \theta \right\}$$

6.3 Graph Traversal

Definition 6.3 (Path Query)

A path query $\pi = r_1 \circ r_2 \circ \cdots \circ r_n$ specifies a pattern of edges. The result of executing $\pi$ from memory $m$ is the set of all memories reachable via matching paths.

  Query --embed--> R^d --ANN--> Candidates
    |                              |
  parse                         filter
    |                              |
    v                              v
  Graph Pattern ---traverse---> Results

6.4 Capability Preservation

Theorem 6.1 (StorageRouter Preserves Capabilities)
$$\mathrm{Cap}(\mathrm{Router}) = \bigcup_{i} \mathrm{Cap}(B_i)$$
Proof.

By the coproduct construction, the router inherits each backend's capabilities through injection morphisms. For any $\mathtt{op} \in \mathrm{Cap}(B_j)$, the router dispatches to $B_j$ via $\iota_j$. Conversely, the router cannot perform unsupported operations since it only delegates.

□

7 Reliability Guarantees

7.1 Vector Clocks for Causal Consistency

Definition 7.1 (Vector Clock)

A vector clock for $n$ agents is $\mathrm{vc}\colon \mathrm{Agents} \to \mathbb{N}$. The partial order is:

$$\mathrm{vc}_1 \leq \mathrm{vc}_2 \iff \forall a.\; \mathrm{vc}_1(a) \leq \mathrm{vc}_2(a)$$

Two clocks are concurrent ($\mathrm{vc}_1 \| \mathrm{vc}_2$) when neither $\leq$ the other.

Each write increments the writer's component:

$$\mathrm{vc}'(a) = \begin{cases} \mathrm{vc}(a) + 1 & \text{if } a = \text{writer} \\ \max(\mathrm{vc}(a), \mathrm{vc}_{\text{received}}(a)) & \text{otherwise} \end{cases}$$

7.2 Content Hashing for Deduplication

$$h(m) = \mathrm{SHA256}(\mathrm{canonical}(m.\mathtt{data}))$$
Proposition 7.1 (Deduplication Soundness)

Content hashing is sound: if $h(m_1) = h(m_2)$, then $m_1.\mathtt{data} = m_2.\mathtt{data}$ with overwhelming probability ($1 - 2^{-256}$).

7.3 Soft Deletes as Filtered Subcategory

Definition 7.2 (Tombstone Subcategory)

The active memory subcategory $\mathcal{M}_{\mathrm{active}} \hookrightarrow \mathcal{M}$ is the full subcategory on memories whose deleted_at field is nil. All queries implicitly filter to $\mathcal{M}_{\mathrm{active}}$, while administrative operations access the full category.

7.4 ACID Properties

Mnesia transactions provide: Atomicity (all-or-nothing), Consistency (schema validation on every write), Isolation (serialized concurrent writes), and Durability (disc_copies to both RAM and disk).

Proposition 7.2 (Transaction Safety)

Every storage operation in the Elixir implementation is wrapped in a Mnesia transaction, ensuring serializability and that partial failures do not corrupt the memory graph.

8.3 Typed Memory GenServer

defmodule MemoryLayer.Memory do
  use GenServer
  alias MemoryLayer.{Schema, Version, Storage}

  @type t(schema) :: %{
    id: String.t(),
    schema_name: String.t(),
    data: schema,
    version: non_neg_integer(),
    content_hash: String.t(),
    created_at: DateTime.t(),
    updated_at: DateTime.t(),
    deleted_at: DateTime.t() | nil
  }

  # --- Public API (Mem[S] interface) ---

  @spec create(schema :: struct()) ::
    {:ok, pid()} | {:error, term()}
  def create(schema_data) do
    GenServer.start_link(__MODULE__, schema_data)
  end

  @spec data(pid()) :: {:ok, struct()} | {:error, term()}
  def data(pid), do: GenServer.call(pid, :get_data)

  @spec evolve(pid(), map(), Version.change_reason()) ::
    {:ok, pid()} | {:error, term()}
  def evolve(pid, changes, reason) do
    GenServer.call(pid, {:evolve, changes, reason})
  end

  # --- GenServer callbacks ---

  @impl true
  def init(schema_data) do
    id = UUID.uuid4()
    now = DateTime.utc_now()
    hash = content_hash(schema_data)

    state = %{
      id: id,
      schema_name: schema_name(schema_data),
      data: schema_data,
      version: 1,
      content_hash: hash,
      created_at: now,
      updated_at: now,
      deleted_at: nil
    }

    # Persist to ETS (working memory)
    :ets.insert(:memory_working, {id, state})
    # Persist to storage backend
    Storage.save(state)

    {:ok, state}
  end

  @impl true
  def handle_call({:evolve, changes, reason}, _from, state) do
    evolved = struct(state.data, changes)
    {:ok, child_pid} = create(evolved)
    {:ok, child_state} = data(child_pid)
    Version.record(state.id, child_state.id, reason)
    {:reply, {:ok, child_pid}, state}
  end
end

8.4 Versioning

defmodule MemoryLayer.Version do
  @type change_reason ::
    :observation | :inference | :correction | :decay

  @spec record(parent_id :: String.t(),
               child_id :: String.t(),
               reason :: change_reason()) :: :ok
  def record(parent_id, child_id, reason) do
    entry = %{
      memory_id: child_id,
      parent_id: parent_id,
      version: next_version(parent_id),
      reason: reason,
      vector_clock: increment_clock(parent_id),
      timestamp: DateTime.utc_now()
    }
    :mnesia.transaction(fn ->
      :mnesia.write({:versions, child_id, entry})
    end)
    :ok
  end

  @spec history(memory_id :: String.t()) :: [version_entry()]
  def history(memory_id) do
    {:atomic, entries} = :mnesia.transaction(fn ->
      :mnesia.match_object({:versions, memory_id, :_})
    end)
    entries
    |> Enum.map(fn {:versions, _, entry} -> entry end)
    |> Enum.sort_by(& &1.timestamp)
  end
end

defmodule MemoryLayer.Graph do

8.6 Storage Backend

defmodule MemoryLayer.Storage do
  @behaviour MemoryLayer.StorageBackend

  # Fan-out save: ETS + Mnesia
  @impl true
  def save(memory) do
    :ets.insert(:memory_working, {memory.id, memory})
    :mnesia.transaction(fn ->
      :mnesia.write({:memories, memory.id, memory})
    end)
    :ok
  end

  # Recall: ETS first (fast path), fallback Mnesia
  @impl true
  def recall(id) do
    case :ets.lookup(:memory_working, id) do
      [{^id, memory}] -> {:ok, memory}
      [] -> recall_from_mnesia(id)
    end
  end

  # Search across backends
  @impl true
  def search(query, opts \\ []) do
    backend = Keyword.get(opts, :backend, :all)
    case backend do
      :exact    -> search_mnesia(query, opts)
      :semantic -> search_embeddings(query, opts)
      :graph    -> search_graph(query, opts)
      :all      -> merge_results([
        search_mnesia(query, opts),
        search_embeddings(query, opts)
      ])
    end
  end
end

8.7 Working Memory vs. Persistent Knowledge

The dual-layer architecture mirrors human cognitive architecture:

  • ETS (Working Memory): in-process, microsecond access, lost on process crash. Holds the "hot" set of memories currently relevant to the agent's task.
  • Mnesia (Persistent Knowledge): distributed, disk-backed, survives restarts. Holds the complete memory store.

The promotion/demotion protocol: on create, written to both ETS and Mnesia; on recall, if in ETS return immediately, otherwise load from Mnesia (promotion); on evict, remove from ETS but retain in Mnesia (demotion). ETS is bounded with LRU eviction.

10 Discussion

Comparison to Unix VFS

The Unix VFS and our StorageRouter share the goal of a uniform interface over heterogeneous backends. However, VFS operates at the byte-stream level; our router operates at the typed-schema level. VFS uses inodes organized in a tree; we use typed instances organized in a labeled graph. VFS supports a fixed set of operations; our protocol is extensible, including semantic search.

Comparison to Plan 9

Plan 9's insight: everything should be a file. Our analogous principle: everything an agent knows should be a typed memory—including execution traces (AGENT_RUN), task queues (TODO), and configuration (CONFIG). Plan 9 used synthetic file servers; we use MCP tools. The key advance: our "files" carry schemas, versions, and graph relationships.

Comparison to Semantic Filesystems

Semantic filesystems pioneered metadata-based retrieval. Our semantic search extends this to meaning-based retrieval via embeddings, with the addition of graph structure and causal versioning.

Relation to Knowledge Graphs

Knowledge graphs typically lack schema-indexed typing (our $\mathsf{Mem}[S]$ enforces strict schemas), causal versioning (our $\mathsf{VMem}[S]$ integrates it as a first-class functor), and embedding integration (our StorageRouter unifies them).

Limitations and Future Work

  1. Garbage collection: the tombstone pattern means monotonic storage growth. A generational GC policy would bound storage while preserving lineage.
  2. Schema evolution: field additions/removals require migration automation.
  3. Scalability: Mnesia has limitations beyond a few hundred nodes; CockroachDB or FoundationDB may be needed at scale.
  4. Embedding drift: model updates invalidate previously computed vectors.
  5. Privacy: the "right to be forgotten" conflicts with tombstones; cryptographic erasure could reconcile this.

11 Related Work

Agent architectures with memory. Soar distinguished working, declarative, procedural, and episodic memory—a taxonomy our 24 types refine. ACT-R formalized activation-based retrieval, inspiring our embedding-based search. Generative agents demonstrated LLM agents benefit from structured memory but lacked type safety and formal versioning.

Type-theoretic databases. Categorical databases (CQL) formalize schemas as categories and instances as functors, closely paralleling our construction. Our contribution adapts this to agent memory with versioning, lineage, and multi-modal storage.

Version control. Git uses content-addressable storage with a DAG of commits. Our version lineage operates on typed memory instances with semantic change-reason annotations.

Distributed consistency. Dynamo popularized vector clocks. Our adaptation applies them to agent memory with the constraint that conflicts must be resolved through typed merge operations.

Memory in LLM systems. MemGPT introduced OS-inspired memory tiers. LangChain's memory modules provide buffer, summary, and entity memory. Our framework subsumes these with formal type guarantees and a categorical foundation.

12 Conclusion

We have presented a categorical framework for agent memory that formalizes the memory layer of an AI Operating System. The key contributions are:

  1. A formal memory model based on the functor $M\colon \mathcal{S} \to \mathcal{S}\!t$, with the Yoneda-lemma connection establishing $\mathsf{Mem}[S]$ as a representable functor (Theorem 3.1).
  2. A type system with 24 schema-indexed memory types and a proven round-trip type-preservation theorem (Theorem 4.1).
  3. A versioning framework based on the temporal functor $V\colon \mathsf{Timeline} \to \mathcal{S}\!t$, with graph lineage as 2-categorical structure and memory operations forming a monad (Proposition 5.1).
  4. A multi-backend storage architecture formalized as a coproduct with proven capability preservation (Theorem 6.1).
  5. A production implementation in Elixir/OTP using ETS and Mnesia, with ACID guarantees and an MCP server for universal client access.
  6. Case studies demonstrating agent learning, cross-session continuity, and multi-agent collaboration.

The memory layer is the third component of the AI Operating System, building on the kernel (Part I) and scheduler (Part II). Together, these provide process management, resource allocation, and persistent knowledge. Part IV will address the tool interface layer.

The categorical perspective provides the compositional guarantees that make the memory layer trustworthy. When an agent stores a decision and later recalls it, the type-preservation theorem ensures conformance. When two agents concurrently update shared knowledge, vector clocks detect conflicts and the monad structure ensures sound resolution. When storage is extended, the coproduct construction preserves existing capabilities.

Agent cognition requires memory that is typed, versioned, searchable, and graph-structured. The filesystem analogy is apt but insufficient: what agents need is not a filesystem but a knowledge system. The categorical memory layer provides exactly this.

D Proof: Monad Laws for Memory Operations (Detailed)

Setup. Let $\mathcal{M}$ be the category of typed memory instances. Define $T\colon \mathcal{M} \to \mathcal{M}$ by:

$$T(m) = (m, \bot, \mathtt{pending})$$

where $\bot$ denotes "no changes yet" and $\mathtt{pending}$ marks the operation as unevaluated. The unit $\eta_m = (m, \emptyset, \mathtt{noop})$ wraps $m$ in a trivial operation. The multiplication $\mu$ collapses $(T(m), \delta_1, r_1)$ with outer $(-, \delta_2, r_2)$ into $(m, \delta_1 \cup \delta_2, r_1 \sqcup r_2)$ where $\sqcup$ is the join in the reason lattice.

Left unit: $\mu_{m} \circ T(\eta_m) = \mu_m((m, \emptyset, \mathtt{noop}), \bot, \mathtt{pending}) = (m, \emptyset, \mathtt{noop}) = T(m)$ after simplification.

Right unit: $\mu_m \circ \eta_{T(m)} = \mu_m(\eta((m, \delta, r))) = \mu_m(((m,\delta,r), \emptyset, \mathtt{noop})) = (m, \delta, r) = T(m)$.

Associativity: For $T^3(m)$ with changes $\delta_1, \delta_2, \delta_3$ and reasons $r_1, r_2, r_3$:

$$\mu \circ T\mu : (m, \delta_1 \cup \delta_2 \cup \delta_3, r_1 \sqcup r_2 \sqcup r_3)$$
$$\mu \circ \mu T : (m, \delta_1 \cup \delta_2 \cup \delta_3, (r_1 \sqcup r_2) \sqcup r_3)$$

These are equal by associativity of $\cup$ and $\sqcup$. □
