# AI Operating System — Knowledge Base

## Consolidated from Exploration of Agent-Hero Ecosystem

### 1. Agent Orchestration (from agent-hero + agent-testing-framework)

**Agent-Hero Execution Pipeline:**
- Job → Proposal → Contract → Decomposition → Execution → Evaluation
- Inngest for durable execution with retry (decompose-job, execute-agent functions)
- Orchestrator decomposes jobs into subtasks with dependency graph (topological sort)
- Executor runs agents with streaming, tool calling (Vercel AI SDK v6 streamText)
- Evaluator scores 6 dimensions: quality, adherence, speed, cost, error, revision
- Oversight modes: supervised, spot_check, autonomous_escalation

**Agent-Testing-Framework 5-Phase Pipeline:**
- Phase 1: Reconnaissance (page discovery, element counting)
- Phase 2: Behavioral Simulation (declarative JSON workflows, per-persona)
- Phase 3: Synthetic Load (concurrent stress testing)
- Phase 4: Adaptive Observation (anomaly detection, correlation)
- Phase 5: Synthesis (aggregation, prioritization, task generation)
- EventEmitter streaming bus — downstream agents consume upstream events in real-time
- Declarative workflow engine with 12 step types and template resolution

**Key Patterns:**
- Separation: Executor (run) vs Evaluator (score) vs Orchestrator (plan)
- Streaming: Real-time output via callbacks + WebSocket
- Fault tolerance: Inngest retries, idempotency guards, abort controllers
- Modularity: Each phase/agent is self-contained with clear interfaces

### 2. Tool Calling & Security (from agent-hero)

**Tool Registry Architecture:**
- 3 tool categories: Builtin (6), Sandbox (6), MCP (runtime-discovered)
- Unified under ToolSet interface for streamText()
- Tool execution traced in execution_trace for audit

**MCP Integration:**
- @modelcontextprotocol/sdk with StreamableHTTPClientTransport
- Parallel connection via Promise.allSettled()
- JSON Schema → Zod conversion for type safety
- Tool namespacing: "${serverName}__${toolName}"

**Security Boundaries:**
- E2B sandboxes for code execution (10-min timeout)
- URL validation: no localhost in production, no internal IPs
- Bearer token auth for MCP servers
- Contract-locked agent configuration (cannot change tools mid-execution)
- Per-action logging with timestamp, input, output

### 3. Typed Memory System (from contextfs)

**Type System (Mem[S]):**
- Generic wrapper for type-safe schema-indexed memory
- BaseSchema → domain schemas (DecisionData, etc.)
- VersionedMem[S] with ChangeReason (Observation, Inference, Correction, Decay)
- SchemaRegistry for runtime type resolution
- 24 memory types across core, extended, workflow categories

**Storage Architecture:**
- StorageBackend Protocol: save(), recall(), search(), delete(), update()
- GraphBackend Protocol: memory lineage (evolve, merge, split)
- Pluggable backends: SQLite (local), ChromaDB (embeddings), FalkorDB (graph), PostgreSQL (cloud)
- StorageRouter coordinates across backends

**Reliability Mechanisms:**
- Vector clocks for conflict resolution
- Content hashing for deduplication
- Soft deletes (tombstone pattern)
- Alembic migrations for schema evolution
- End-to-end encryption support

**Key Patterns:**
- Layered: Clients → ContextFS core → StorageRouter → Backends
- Type safety: Compile-time (mypy) + runtime (pydantic)
- Cross-device: Portable paths + vector clocks for sync
- Versioning: Every change tracked with causality

### 4. Planning & Market Dynamics (from agent-hero)

**Marketplace Structure:**
- Roles: client (posts jobs), operator (owns agents), both
- Job lifecycle: draft → open → in_progress → review/completed/cancelled/disputed
- Proposal lifecycle: pending → shortlisted/rejected/accepted
- Contract lifecycle: active → paused/completed/cancelled/disputed

**Bidding Mechanics:**
- Operator submits: execution_plan, estimated_credits, estimated_duration, confidence_score
- Client accepts one → all others auto-rejected
- Acceptance creates Contract with escrow hold

**Financial System (Order Book analogy):**
- Escrow: holdEscrow(client_id, rate_credits, contract_id)
- Revenue split: 70% operator, 15% platform, 15% LLM reserve
- Atomic operations via Supabase RPC (prevents race conditions)
- 3 pricing models: per_task, hourly, per_token
- Budget ceiling prevents overspend

**Orchestration:**
- decomposeAndExecute() → Inngest event → LLM decomposes into subtasks
- Topological sort of dependencies
- Parallel execution with dependency barriers
- Quality evaluation triggers reputation recomputation

### 6. Why Erlang/Elixir for AI OS

- BEAM VM designed as telecom OS — process scheduling, fault tolerance, hot reload
- OTP Supervisors → Agent Supervisors (restart strategies)
- GenServer → Agent process (state + message handling)
- ETS → Working memory (in-process, concurrent reads)
- Mnesia → Persistent knowledge (distributed, replicated)
- Registry → Service discovery (agent lookup)
- "Let it crash" → Agent fault tolerance
- Lightweight processes (millions) → Agent scalability
- Message passing → Agent communication (no shared state)
- Distribution → Multi-node agent deployment
